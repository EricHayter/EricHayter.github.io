yadb

A relational database engine built from scratch in C++20. Bison/Flex SQL parser, Volcano-model executor, slotted-page storage with LRU-K buffer pool and async disk I/O.

View source

What it is

yadb is a relational database engine built from scratch in C++20. The goal was to understand every layer — from SQL text on input to bytes flushed to disk — without borrowing existing components.

Query processing

A Bison/Flex lexer and parser turns SQL text into an AST. A system catalog stores table and column metadata. The executor uses a Volcano-model pull interface: FileScan, Projection, Selection, and HashJoin each implement a single next() method.

// yadb/src/operator.h
class FileScan : public Operator {
  BufferPool& pool_;
  PageId      cur_{0};
public:
  auto next() -> std::optional<Tuple> {
    auto pg = pool_.pin(cur_++);
    return pg.read_next_tuple();
  }
};

Storage

Under the executor lives a storage engine built around a slotted-page format. An LRU-K buffer pool manages the frame cache. Asynchronous disk I/O decouples page fetches from query execution so the buffer pool doesn't stall on every cache miss.

What I learned

  • Slotted pages are the right default: variable-length tuples without fragmentation headaches.
  • LRU-K over plain LRU matters most under scan-heavy workloads that would otherwise evict hot pages.
  • Volcano looks simple until you need to push predicates down — then the clean pull abstraction fights you.
